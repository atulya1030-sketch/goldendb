GoldenDB

Recruiter intelligence platform for AI/data talent in India — built on a vector database, powered by GPT-4o, with automated candidate sourcing from GitHub and Apollo.

What is GoldenDB

GoldenDB is a talent intelligence platform built specifically for recruiting AI and data engineering professionals in India. It replaces the traditional recruiter workflow of manually screening LinkedIn profiles and spreadsheets with a conversational AI layer on top of a deeply structured candidate database.

Every candidate in GoldenDB has a rich profile — not just name and title, but technical depth scores, system-level signals (what they've actually built), behavioural calibration, compensation bands, retention probabilities, and pipeline history. The AI can reason across all of these fields to answer complex recruiting questions in seconds.

There are two sides to the platform: client-facing (what recruiters and hiring managers see) and internal (how GoldenDB staff source and build the candidate pool).

Candidate Pipeline

Every candidate moves through four stages. Only stage 4 candidates are visible to clients. Stages 1–3 are internal work-in-progress.

  • Stage 1 (sourced): Candidate discovered via GitHub search or Apollo. We have their GitHub handle, email, phone, LinkedIn, and whatever public data GitHub exposes. No resume yet.
  • Stage 2 (resume_parse): Resume PDF uploaded. GPT-4o extracts all structured fields — title, company, skills with scores, systems built, compliance exposure, education, compensation expectations.
  • Stage 3 (enriched): Cross-platform enrichment run. GitHub stars/followers added, Kaggle competition tier noted, Stack Overflow reputation and top tags pulled, HuggingFace models and downloads logged. Depth and composite scores updated.
  • Stage 4 (panel_complete): Internal panel interview completed. All scores are verified by a human. Candidate is client-ready and searchable in the recruiter chat.

Client-facing
Recruiter Chat
A conversational interface where recruiters and hiring managers search the verified candidate pool. Powered by GPT-4o with three specialised tools: semantic candidate search, market aggregation, and full profile lookup. Responses stream in real time via SSE.

What it can do

  • Find candidates by role, city, skills, notice period, budget, compliance exposure, or any combination
  • Build teams — "give me a 4-person data team: staff DE, ML engineer, analytics lead, MLOps"
  • Answer market questions — "which Indian city has the most relocatable fintech engineers?"
  • Surface hidden gems — "who undersells on paper but scored high in panel?"
  • Every result links to the full candidate profile page

Why we built it this way

Recruiters don't think in SQL. They think in natural language — "senior engineer, Bengaluru, not FAANG-locked, can join in 30 days." A chat interface lets them express exactly that. The underlying vector search + hard filters combination means semantic intent AND hard constraints are both respected simultaneously. The AI relaxes filters automatically when results are too narrow and always explains what it changed.
/source-chat
Internal only
Sourcing Copilot
A conversational sourcing tool for the GoldenDB team to discover new candidates from GitHub. Type what kind of engineer you're looking for, and the AI searches GitHub, scores every profile, enriches with Apollo contact data (email, phone, LinkedIn), and presents results you can save directly into the pipeline.

What it can do

  • Search GitHub by role, city, skills, seniority in natural language
  • Score up to 50 profiles per query on followers, stars, location, activity, language match
  • Return top 15 with Apollo-enriched contact info (email, phone, LinkedIn)
  • Flags guessed emails (not verified) so you know before you outreach
  • Save any or all candidates to GoldenDB with one message ("save the top 3")
  • Refine in conversation — "remove FAANG", "only score 12+", "now find similar in Mumbai"
  • Saved candidates enter the pipeline at stage 1 and work their way to client-ready

Why we built it this way

GitHub is the most honest signal for an AI/data engineer — what they've actually built, in public, over years. LinkedIn tells you job titles. GitHub tells you depth. The scoring system weights followers, stars, recent activity, and language match so we spend Apollo enrichment credits only on genuinely strong profiles. A chat interface lets the sourcer iterate quickly without filling out forms.
/source
Internal only
Form-Based Sourcing
The same sourcing pipeline as /source-chat but presented as a traditional search form. Type a query, hit Search, and candidate cards stream in one by one as GitHub profiles are fetched and Apollo contact data is added. Each card has an "Add to GoldenDB" button.

What it can do

  • Same GitHub search + scoring + Apollo enrichment as the sourcing chat
  • Results stream in progressively — cards appear as they're processed
  • Score badge on each card (Strong / Good) so you can quickly skip weak profiles
  • Contact info shown inline: email (with verified/guessed status), phone, LinkedIn, GitHub
  • "Save all" button to bulk-add the entire result set to the pipeline

Why we built it this way

Some users prefer a visual card layout over a chat. This gives the same sourcing power with a more structured, scannable output. The two sourcing interfaces (/source and /source-chat) serve the same function — use whichever suits you.
/upload
Internal only
Resume Ingest
Upload a candidate's resume PDF and GoldenDB extracts all structured fields automatically using GPT-4o. Optionally provide their GitHub, Kaggle, Stack Overflow, or HuggingFace handles to trigger cross-platform enrichment immediately after ingest.

What it can do

  • Accept any resume PDF, extract text with pdfplumber, send to GPT-4o for structured extraction
  • Extracts: name, title, company, skills with scores and years, systems built, compliance exposure, education, notice period, compensation expectations, and more
  • Auto-computes tier (T2–T6), composite score, retention estimate from extracted data
  • Optionally runs GitHub, Kaggle, Stack Overflow, and HuggingFace enrichment in the same step
  • Shows a result card preview before saving — skills bars, top systems, key stats
  • Links to the full profile page once saved

Why we built it this way

Manually entering candidate data into a schema is error-prone and slow. GPT-4o can extract a well-structured JSON payload from any resume format in under 10 seconds. This takes a candidate from "PDF in email" to "searchable in the vector database" with no manual data entry. Combined with optional enrichment, a candidate can go from stage 1 to stage 3 in a single upload session.
/profile/:candidate_id
Client-facing
Candidate Profile Page
A full candidate profile rendered from live data. Every link in the recruiter chat (e.g. /profile/GDB-1041) lands here. Two tabs: Client Profile (what the client sees) and Internal Signals (recruiter-only intel). Includes infographics, scoring breakdowns, and panel mechanics.

What it contains

  • Client Profile tab: Identity, availability/comp/mobility, technical depth, systems built, compliance exposure, education, behavioural calibration, motivation fit, and predictive scores
  • Internal Signals tab: Pipeline ops, internal score estimates, risk/timing/conflict flags, psychology calibration, panel mechanics widget (two simulated panelist scores), and scoring infrastructure details
  • Infographics: Readiness gauge, depth journey chart (v1→v2→v3 progression), skill bars for top 6 skills
  • State chips: Every field is tagged — ✓ VER (verified by panel), ● UNVER (unverified), ⚙ SYS (system-computed), 🔒 LOCKED (panel-gated), — (null-valid field)

Why we built it this way

A recruiter needs to walk into a client meeting able to speak confidently about every candidate. The profile page gives them everything in one place — not just skills and experience but predicted retention, compensation delta tolerance, management aspiration, red flags, and panel scores. The two-tab design separates what clients need from the internal intelligence that drives sourcing decisions.

  • candidate_id (GDB-XXXX)
  • full_name, current_title
  • current_company, company_tier
  • role_family, location_city
  • hometown_city, hometown_pull

Experience

  • tier (T2–T6)

  • overall_band (gold/silver/bronze)
  • composite (0–100)
  • growth_velocity

Technical

  • skills (dict: score, years, recency)
  • systems_built (enum list)
  • compliance (enum list)
  • scale_tb, migration
  • genai_production
  • oss_maintainer

Education

  • degree, institution

  • grad_year
  • accolades, teaching
  • community_roles

Availability & Comp

  • notice_days, notice_negotiable
  • ctc_current, ctc_expected_min/max
  • search_status, work_mode
  • timezone_flex
  • intl_exposure

Behavioural

  • leadership_aspiration, ceiling
  • builder_archetype, chaos_tolerance
  • motivation_type, fit_context
  • ownership, exec_exposure
  • communication, resilience

Predictive

  • retention_6m, retention_12m
  • likelihood_to_move_6m
  • resume_panel_delta

Why vector search instead of SQL

A recruiter asking "deliberate IC who refused management and built data lakes at scale" cannot be answered with SQL. It's a semantic query — the meaning matters, not keyword matching. Qdrant stores a 384-dimensional embedding of each candidate's full profile bio, so semantic queries find the right people even when the exact words don't match. Hard constraints (city, notice, budget) are applied as filters on top of the vector search, combining meaning with precision.
